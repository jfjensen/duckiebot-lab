# duckiebot-lab

The **application layer** for the pyhut Duckiebot project. `pyhut` is the
hardware-agnostic HAL (standard-library only); everything that is *policy*,
*experiment*, or *off-robot plumbing* lives here and depends on `pyhut` one-way.

This repo is the home for the Stage 4 motion primitives, the Stage 5 lidar demo,
and the Stage 6 telemetry. The first thing to land is the **Stage 2b motor-EMI
diagnostic**.

## What the motor-EMI diagnostic answers

The motor board (PCA9685 + TB6612 + the motor leads/coils) sits on the same HUT
as the AK8963 magnetometer, so **spinning the motors can corrupt the heading**.
Stages 4/5 do an in-place 360° spin to build a "lidar" scan and want to tag each
range with a heading. This experiment decides one thing:

> Is the magnetometer usable as a **live** heading input *during* the spin, or
> only as a **static** before/after reference?

The answer feeds Stage 4's sensor-fusion decision (fuse the mag mid-spin only if
this run says it's usable).

## How it works, briefly

The mag measures the field in the sensor frame:

```
measured(θ) = R(θ)·B_earth  +  b_emi
```

Earth's field magnitude is constant — turning the robot only rotates the vector.
The EMI term `b_emi` is generated by motor current; the leads are fixed relative
to the mag, so for a constant motor state it's a roughly constant sensor-frame
vector whose size grows with duty. Two levers separate EMI from real rotation:

1. **Bench mode (recommended):** put the robot on a stand with the wheels free.
   The chassis doesn't turn, so `measured_on − measured_off = b_emi` directly,
   per duty. Read the induced heading error straight off it.
2. **|B|-invariance (works on the floor too):** real rotation preserves the
   field magnitude; EMI does not. A `|B|` change that tracks duty is EMI. The
   **gyro** is the cross-check — near-zero gyro confirms bench mode; a spinning
   gyro means the raw heading error is contaminated by genuine turning and the
   tool leans on the `|B|` / worst-case columns instead.

The planning number is the **worst-case heading error**, `asin(|EMI_h| / |H|)` —
the error that occurs when the EMI vector is perpendicular to the horizontal
Earth field `H`, which the in-place spin sweeps through on every turn.

**Repeat the operating duty.** On a weak or noisy field the EMI at a single duty
can be comparable to the sensor noise, so one measurement isn't enough — the
same spin can read low once and high the next time. Repeating a duty (e.g.
`--duties 0.5,0.5,0.5,0.5,0.5`) exposes that: the verdict keys off the **worst**
error observed at each duty, and a wide spread across identical repeats is itself
decisive evidence the mag can't be trusted live. A live heading input you can't
reproduce from one spin to the next is unusable by definition, whatever its
average looks like.

## Install

`pyhut` is the one-way dependency. It isn't on PyPI, so install it first (or let
the git reference in `pyproject.toml` pull it):

```bash
# Option A: install pyhut from its repo, then this package.
pip install git+https://github.com/jfjensen/pyhut.git
pip install -e .

# Option B (local dev): editable installs side by side.
pip install -e ../pyhut
pip install -e .            # the git dep resolves as already-satisfied

# Optional off-robot extras (PNG plot, tests):
pip install -e ".[plot,test]"
```

The on-robot runtime is standard-library and Python 3.6-safe, so nothing extra
has to be installed on the stock JetPack image to run a diagnostic. `matplotlib`
is optional and only used for the PNG plot off-robot.

### Installing on the Jetson (Python 3.6)

The stock JetPack image ships Python 3.6 with an old pip/setuptools that can't
build from `pyproject.toml` alone (`pip install -e .` fails with *"File
'setup.py' not found"* or installs a blank `UNKNOWN 0.0.0` package). A small
`setup.py` shim carries the metadata for that toolchain:

```python
from setuptools import setup, find_packages

setup(
    name="duckiebot-lab",
    version="0.1.0",
    package_dir={"": "src"},
    packages=find_packages(where="src"),
    python_requires=">=3.6",
    extras_require={"test": ["pytest<7.1"], "plot": ["matplotlib"]},
    entry_points={
        "console_scripts": [
            "duckiebot-emi = duckiebot_lab.experiments.motor_emi:main",
        ],
    },
)
```

Then `pip3 install -e .` registers the package and the `duckiebot-emi` command.
`pyproject.toml` stays the source of truth for modern toolchains; `setup.py` is
only the shim old setuptools needs. `pytest<7.1` is the last line with solid 3.6
support. To just run the tests without installing, `PYTHONPATH=src python3 -m
pytest` works too — and note bare `pytest` on the Jetson may be Python 2's, so
use `python3 -m pytest`.

## Run it

On the robot (as root, since pyhut talks to `/dev/i2c-1` and sysfs GPIO):

```bash
# Cleanest result: robot on a stand, wheels free (bench mode).
sudo duckiebot-emi --calibration ../pyhut_calibration.json --plot

# If Stages 4/5 will spin at a known duty, pin the pass/fail to it:
sudo duckiebot-emi --operating-duty 0.5 --calibration ../pyhut_calibration.json

# Confirmation run: repeat the operating duty to see the run-to-run spread.
sudo duckiebot-emi --operating-duty 0.5 --dwell 12 \
    --duties 0.5,0.5,0.5,0.5,0.5 --calibration ../pyhut_calibration.json
```

If `duckiebot-emi` isn't found under `sudo` (it installs to `~/.local/bin`,
which root's PATH may not include), use `sudo $(which duckiebot-emi) …` or
`sudo python3 -m duckiebot_lab.experiments.motor_emi …`.

Pass `--calibration` with the same pyhut calibration Stages 4/5 use, so the run
samples through the *calibrated* view they'll consume and the reported heading
error is what they'd actually see. **Check the path resolves to a real file** —
an uncalibrated run reads a large hard-iron offset as the field (e.g. `|B|`
~110 µT) and gives a falsely clean result. A sane calibrated baseline has
`|B| ≥ |H|`; if the report shows `|B| < |H|`, the calibration is bad (often an
under-swept Z axis) — re-run `pyhut.calibrate` where the robot actually operates
before trusting anything.

Try it **without hardware** (synthetic robot, for the output shape and a demo):

```bash
duckiebot-emi --simulate --sim-mode bench
duckiebot-emi --simulate --sim-mode floor    # chassis really spins; gyro trips
```

### Useful flags

- `--duties 0.1,0.2,...` — the motor duties to sweep; repeat a value (e.g.
  `0.5,0.5,0.5`) to measure run-to-run spread at that duty.
- `--pattern spin|straight` — in-place spin (default) or straight drive.
- `--dwell / --settle / --baseline` — timing per window (seconds).
- `--threshold` — worst-case heading-error budget (default 5°).
- `--operating-duty` — the duty Stages 4/5 will spin at; pins the verdict.
- `--no-rebaseline` — skip the off-window between every level (faster, less drift-robust).
- `--outdir / --stem` — where the run is saved.

## What you get

A clear verdict plus the numbers behind it, printed and saved to `--outdir`:

- `*.json` — the canonical record. Its `verdict` block is machine-readable
  (`status`, `mag_live_usable`, `max_safe_duty`, …) so **Stage 4 can read it
  directly**.
- `*.csv` — the sweep table (duty vs EMI magnitude, heading error, |B| deviation, gyro RMS).
- `*.txt` — the human report.
- `*.png` — optional plot (`--plot`, needs matplotlib).

### The three verdicts

- **LIVE_OK** — worst-case error stays under threshold across the tested range:
  fuse the mag live during the spin.
- **MARGINAL** — under threshold at low duty, over it higher up: live-usable only
  if the spin is kept at/under the reported `max_safe_duty`.
- **STATIC_ONLY** — over threshold even at the lowest duty: use the mag as a
  static before/after reference — hold the last good heading through the turn and
  re-anchor after.

Two things the verdict logic does deliberately, both biased toward *not* handing
out a false green light:

- **Worst-of-repeats.** When a duty is measured more than once, the verdict uses
  the worst error seen at that duty, not an average or a lucky sample. One
  over-budget repeat disqualifies the duty.
- **One-directional noise gate.** A tiny effect measured against a *tight*
  at-rest noise floor is forgiven (not clearly distinguishable from noise). But a
  *large* noise floor never upgrades the verdict — a magnetometer that is already
  noisy at rest is worse, not safer. When the at-rest heading noise is a large
  fraction of the budget, the report says so explicitly: treat the mag as a
  coarse heading reference even before EMI.

### Result on this unit (Stage 2b outcome)

On the current robot the verdict is **STATIC_ONLY**: the magnetometer is **not**
usable as a live heading input during the in-place spin. At the operating duty
(0.5), repeated spins gave worst-case heading errors of roughly 3–7° against a 5°
budget, with a ~4° spread across identical repeats. The drivers:

- Weak horizontal field here (`|H|` ~12 µT, vs a typical 20–40 µT), so small EMI
  vectors translate into large heading errors.
- Motor field that isn't even stable run to run (`|B|` deviation swinging roughly
  −8% to +12% at a fixed duty).
- A coarse mag to begin with — ~3° at-rest heading noise even with motors off.

**Consequence for Stage 4:** during the lidar spin, carry heading on the gyro and
do not fuse the mag. Use the mag only for a static before/after anchor to bound
gyro drift, and treat even that as coarse (±3° at best). The machine-readable
`verdict` block in the saved JSON encodes this for the fusion code to read.

## Hardware notes

- The left and right drive paths are electrically asymmetric (left = all PCA9685
  channels; right = GPIO direction + PCA PWM), and an in-place spin runs one
  motor forward and one in reverse, so `spin` is characterized as its own
  pattern rather than assumed equal to straight driving.
- This run **requires working motors**. On the unit with the known open-contact
  fault (motor PCA9685 `0x60` absent when direct-seated), run it over the 40-pin
  ribbon so the motor board enumerates.
- pyhut's drivetrain has a 0.3 s watchdog; the experiment re-asserts the target
  continuously while driving, and always stops the wheels in a `finally` block.

## Tests

```bash
pip install -e ".[test]"
pytest                       # or: python3 -m pytest   (use this on the Jetson)
```

The suite (16 tests) runs entirely against a synthetic robot (`_sim.py`) that
implements pyhut's ABCs and injects a physically-shaped EMI model — no hardware
needed. It covers the math helpers, EMI-vector recovery, each verdict branch, the
bench/floor end-to-end paths, and regression tests for the verdict-safety rules
(worst-of-repeats and the one-directional noise gate) grounded in real
on-hardware runs.
